## 왜 필요합니까?

첫 출근한다고 상상해보십시오. 코드베이스에 파일이 수천 개입니다. 다 읽으려면 몇 주가 걸릴 것입니다. 그런데 오늘 당장 버그를 고쳐야 합니다.

이럴 때 Claude로 코드 탐색이 빛을 발합니다. 폴더를 일일이 뒤지는 대신, Claude에게 필요한 것을 찾아달라고 하면 됩니다.

**실제 상황 예시:**
- 프로젝트를 인수받았는데 어떻게 동작하는지 이해해야 함
- 어딘가에 버그가 있는데 어떤 파일을 봐야 할지 모름
- 기능을 추가하고 싶은데 참고할 비슷한 코드를 찾아야 함
- 다른 사람 코드를 리뷰하는데 맥락을 이해해야 함

**Claude를 코드 관광 가이드라고 생각하십시오:**
- Claude 없이: 낯선 거리를 돌아다니며 식당 찾기
- Claude와 함께: 모든 것이 어디 있는지 정확히 아는 동네 친구와 함께

---

## 왜 코드 탐색이 중요합니까?

새로운 프로젝트를 시작하면 "이 코드가 뭐하는 거지?" 궁금할 때가 많습니다.

직접 코드를 다 읽을 필요 없습니다. Claude에게 물어보면 핵심을 파악해서 설명해줍니다.

**효과적인 탐색 요청:**

```
> 이 프로젝트에서 인증 관련 코드 찾아줘
> src/utils 폴더가 뭐 하는 곳이야?
```

구체적인 기능이나 폴더를 지정하면 더 정확한 답을 얻습니다.

---

## 프로젝트 구조 파악하기

### 전체 구조 물어보기

```
> 이 프로젝트 구조 설명해줘
```

```
> 이 프로젝트가 뭐 하는 프로젝트야?
```

```
> 어떤 기술 스택 쓰고 있어?
```

Claude가 폴더 구조와 주요 파일들을 분석해서 설명해줍니다.

### 특정 폴더 살펴보기

```
> @src/ 이 폴더 안에 뭐가 있어?
```

```
> components 폴더 역할이 뭐야?
```

## 파일 찾기 (Glob)

Glob은 이름 패턴으로 파일을 찾는 방법입니다.

### 자주 쓰는 패턴

| 요청 | Claude가 하는 일 |
|------|-----------------|
| "js 파일 다 찾아줘" | `*.js` 패턴으로 검색 |
| "테스트 파일 찾아줘" | `*.test.js` 또는 `*.spec.js` 검색 |
| "설정 파일 찾아줘" | `*.config.*`, `*.json`, `*.yaml` 검색 |

### 예시

```
> src 폴더에서 js 파일 다 찾아줘
```

```
> 테스트 파일들 어디 있어?
```

```
> package.json 파일 찾아줘
```

### 패턴 문법 (참고용)

직접 외울 필요는 없습니다. 자연어로 요청하면 Claude가 적절한 패턴으로 변환합니다.

| 패턴 | 의미 |
|------|------|
| `*.js` | js로 끝나는 파일 |
| `**/*.ts` | 모든 폴더의 ts 파일 |
| `src/**/*` | src 안의 모든 파일 |

## 코드 검색하기 (Grep)

Grep은 파일 안에서 특정 텍스트를 찾는 방법입니다.

### 자주 쓰는 검색

```
> "useState" 쓰는 파일 찾아줘
```

```
> "handleSubmit" 함수 어디 정의돼 있어?
```

```
> TODO 주석 다 찾아줘
```

### 함수 사용처 찾기

```
> fetchUser 함수 어디서 호출해?
```

Claude가 해당 함수가 사용된 모든 파일과 위치를 찾아줍니다.

### API 엔드포인트 찾기

```
> API 엔드포인트 다 찾아줘
```

```
> /api/로 시작하는 경로 찾아줘
```

## 탐색 워크플로우

새 프로젝트를 이해하는 순서:

### 1단계: 전체 그림 파악

```
> 이 프로젝트가 뭐야? 간단히 설명해줘.
```

### 2단계: 구조 이해

```
> 폴더 구조 보여주고 각 폴더가 뭐하는지 알려줘.
```

### 3단계: 진입점 찾기

```
> 이 앱이 어디서 시작해? 메인 파일이 뭐야?
```

### 4단계: 관심 있는 부분 탐색

```
> 로그인 기능 어떻게 구현돼 있어?
> 관련 파일 찾아서 설명해줘.
```

## Plan 모드로 안전하게 탐색

파일을 바꾸지 않고 탐색만 하고 싶을 때 Plan 모드를 사용하십시오.

```
> /plan
> 이 프로젝트 분석해줘. 수정하지 말고 설명만.
```

Plan 모드에서는:
- 파일 읽기: 가능
- 파일 수정: 불가능
- 명령어 실행: 불가능

안전하게 탐색할 수 있습니다.

## 실습해보기

### 실습 1: 오픈소스 탐색

GitHub에서 아무 프로젝트나 클론해서 탐색해보십시오.

```bash
git clone https://github.com/expressjs/express.git
cd express
claude
```

```
> /plan
> 이 프로젝트 구조 설명해줘
> 주요 파일이 뭐야?
> 이 프로젝트로 뭘 할 수 있어?
```

### 실습 2: 파일 찾기

```
> 테스트 파일 다 찾아줘
> 설정 파일들 어디 있어?
> README 파일 찾아줘
```

### 실습 3: 코드 검색

```
> "export" 쓰는 파일 찾아줘
> "TODO" 주석 다 찾아줘
> "error" 단어 포함된 줄 찾아줘
```

## 미니 프로젝트: 코드베이스 분석 보고서

오픈소스 프로젝트를 분석해서 보고서를 만들어보십시오.

### 목표

- 실제 프로젝트 구조 이해하기
- 코드 탐색 능력 기르기

### 프로젝트 선택

```bash
# 추천 프로젝트 (규모별)
git clone https://github.com/sindresorhus/ora.git      # 작음
git clone https://github.com/expressjs/express.git    # 중간
git clone https://github.com/facebook/react.git       # 큼
```

### 분석하기

```
> /plan
> 이 프로젝트를 분석해서 보고서 만들어줘:
> 1. 프로젝트 목적
> 2. 기술 스택
> 3. 폴더 구조
```

### 심화 분석 (숙련자용)

```
> 테스트 커버리지 어떻게 구성되어 있어?

> 성능 병목이 될 만한 부분 찾아줘
```

## 문제가 발생하면

**Claude가 분명 있는 파일을 못 찾습니까?**
- 올바른 디렉토리에 있는지 확인하십시오
- 더 구체적으로: "src/ 아래에서 이름에 'auth' 포함된 파일 찾아줘"
- 파일 이름이 예상과 다를 수 있습니다

**Claude가 결과를 너무 많이 보여줍니까?**
- 범위 좁히기: "components 폴더 안의 테스트 파일만 찾아줘"
- 필터 추가: ".js 말고 .ts 파일만 찾아줘"

**Claude가 "프로젝트에 접근할 수 없다"고 합니까?**
- 프로젝트 폴더 안에서 Claude Code를 실행하고 있는지 확인하십시오
- 시도: `cd /path/to/project && claude`

**탐색이 너무 오래 걸립니까?**
- 범위 제한: "src 폴더만 봐줘"
- "모든 것 설명해줘" 대신 구체적인 질문하기

## 자주 하는 실수

### 실수 1: 한 번에 모든 것 이해하려 하기
```
# 나쁜 예
> 이 프로젝트의 모든 파일 설명해줘

# 좋은 예
> 먼저 폴더 구조 보여줘.
> 그 다음에 어떤 부분 탐색할지 고를게.
```

### 실수 2: 안전한 탐색에 Plan 모드 안 쓰기
```
# 위험 - Claude가 발견한 것들을 "고치려" 할 수 있음
> 이 프로젝트 분석하고 문제 있으면 고쳐줘

# 안전 - 읽기만
> /plan
> 이 프로젝트 분석해줘. 설명만 하고 아무것도 바꾸지 마.
```

### 실수 3: 모호한 검색 요청
```
# 나쁜 예
> 함수 찾아줘

# 좋은 예
> "handleLogin" 함수가 어디 정의되어 있는지 찾아줘
```

### 실수 4: 프로젝트 맥락 무시하기
```
# 나쁜 예 (Python 프로젝트에서)
> .js 파일 다 찾아줘

# 좋은 예
> 먼저 이게 어떤 종류 프로젝트인지 알려줘.
> 그 다음에 주요 소스 파일 찾아줘.
```
