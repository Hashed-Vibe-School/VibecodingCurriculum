## Git이 무엇입니까?

Git은 코드의 버전을 관리하는 도구입니다.

### 왜 필요합니까?

- **되돌리기**: 뭔가 잘못되면 이전 버전으로 돌아갈 수 있습니다
- **협업**: 여러 사람이 같은 코드를 동시에 작업할 수 있습니다
- **기록**: 누가 언제 뭘 바꿨는지 알 수 있습니다

**Git은 코드를 위한 타임머신이라고 생각하십시오:**
- Git 없이: 중요한 코드를 실수로 삭제. 영원히 사라짐.
- Git과 함께: 시간을 되돌려서 몇 초 만에 복구.

**또는 비디오 게임의 여러 세이브 파일처럼:**
- 각 커밋 = 돌아갈 수 있는 세이브 포인트
- 각 브랜치 = 탐험할 수 있는 다른 스토리라인
- 망치면 = 마지막 세이브 불러오기

### 기본 개념

| 용어 | 뜻 |
|------|-----|
| **Repository (저장소)** | Git으로 관리되는 프로젝트 폴더 |
| **Commit** | 변경 사항을 저장하는 것 (세이브 포인트) |
| **Branch** | 독립적인 작업 공간 |
| **Push** | 내 컴퓨터 → 서버로 업로드 |
| **Pull** | 서버 → 내 컴퓨터로 다운로드 |

## GitHub이 무엇입니까?

**GitHub**은 Git 저장소를 온라인에 저장하고 공유하는 클라우드 서비스입니다.

### Git vs GitHub

| Git | GitHub |
|-----|--------|
| 내 컴퓨터의 도구 | 웹사이트 (github.com) |
| 버전 기록 관리 | 코드를 온라인에 저장 |
| 오프라인 작업 가능 | 협업 가능 |
| 무료 소프트웨어 | 무료 + 유료 플랜 |

**비유**: Git은 카메라(사진 찍기), GitHub은 사진 공유 사이트(저장하고 공유)와 같습니다.

### GitHub에서 할 수 있는 것

- **백업**: 코드를 클라우드에 저장
- **협업**: 다른 사람들과 같은 프로젝트 작업
- **오픈소스**: 코드를 공개적으로 공유
- **Pull Request**: 코드를 합치기 전에 리뷰 요청
- **Issues**: 버그와 기능 요청 추적

### GitHub에 연결하기

```
> 이 프로젝트를 GitHub에 연결해줘
```

Claude가:
1. GitHub 로그인 여부 확인
2. 필요시 저장소 생성
3. 코드 푸시

## 커밋하기

커밋은 "현재 상태를 저장"하는 것입니다. 게임의 세이브 포인트와 같습니다.

### Claude에게 커밋 시키기

```
> 지금까지 변경한 거 커밋해줘
```

Claude가:
1. 뭐가 바뀌었는지 확인 (`git status`)
2. 변경 내용 분석 (`git diff`)
3. 적절한 커밋 메시지 작성
4. 커밋 실행

**더 좋은 커밋 메시지를 원한다면:**

```
> 변경사항 커밋해줘. Conventional Commits 형식으로.
> 이번 변경은 버그 수정이야.
```

맥락을 알려주면 더 정확한 메시지가 나옵니다. "새 기능", "버그 수정", "리팩토링" 같은 힌트를 주십시오.

### 커밋 메시지 형식

Claude는 이런 형식으로 커밋 메시지를 씁니다:

```
feat: 로그인 기능 추가

- 이메일/비밀번호 입력 폼 생성
- 로그인 API 연동
```

**타입 종류**:
| 타입 | 의미 |
|------|------|
| `feat` | 새 기능 |
| `fix` | 버그 수정 |
| `docs` | 문서 변경 |
| `style` | 코드 스타일 (동작 변화 없음) |
| `refactor` | 리팩토링 |

## 상태 확인하기

### 현재 상태 보기

```
> git 상태 확인해줘
```

Claude가 `git status`를 실행해서 보여줍니다:
- 수정된 파일
- 새로 추가된 파일
- 커밋 안 된 변경사항

### 변경 내용 보기

```
> 뭐가 바뀌었는지 보여줘
```

Claude가 `git diff`로 변경 내용을 보여줍니다.

## 브랜치 사용하기

브랜치는 "작업 공간"입니다. 메인 코드에 영향을 주지 않고 따로 작업할 수 있습니다.

### 브랜치 만들기

```
> 로그인 기능 만들 브랜치 만들어줘
```

Claude가 `git checkout -b feature/login` 같은 명령어를 실행합니다.

### 브랜치 확인

```
> 지금 어떤 브랜치야?
```

```
> 브랜치 목록 보여줘
```

### 브랜치 전환

```
> main 브랜치로 이동해줘
```

## Pull Request (PR) 만들기

PR은 "내 변경사항을 메인에 합쳐달라"고 요청하는 것입니다.

### Claude에게 PR 시키기

```
> 이 브랜치로 PR 만들어줘
```

Claude가:
1. 변경사항 분석
2. PR 제목과 설명 작성
3. GitHub에 PR 생성

### PR 형식

```markdown
## Summary
- 로그인 폼 구현
- 비밀번호 검증 추가

## Test Plan
- [x] 로그인 성공 테스트
- [x] 잘못된 비밀번호 테스트
```

---

## 충돌 해결하기

여러 사람이 같은 파일을 수정하면 "충돌"이 날 수 있습니다.

### 충돌이 나면

```
> 머지 충돌 났어. 해결해줘.
```

Claude가:
1. 각 버전이 뭔지 설명
2. 어떻게 합칠지 제안
3. 수정 적용

**충돌 해결 요청 팁:**

```
> 충돌 해결해줘. 내 브랜치의 로그인 로직을 유지하고 싶어.
> 두 버전 모두 필요해. 합쳐줘.
```

어떤 버전을 선호하는지, 또는 둘 다 필요한지 알려주면 의도에 맞게 해결합니다.

## Git 안전 규칙

Claude는 위험한 git 명령어 전에 물어봅니다.

### 안전한 명령어 (자동 실행 가능)

- `git status` - 상태 확인
- `git diff` - 변경 내용 보기
- `git log` - 기록 보기
- `git add` - 파일 추가
- `git commit` - 커밋하기

### 위험한 명령어 (확인 필요)

- `git push` - 서버로 업로드
- `git push --force` - 강제 푸시 (매우 위험)
- `git reset --hard` - 변경사항 삭제 (복구 불가)

## 따라해보기

지금 바로 Claude와 Git을 연습해봅시다:

### 연습 1: 첫 번째 저장소

```
# 새 폴더 만들고 Git 초기화
> git-practice라는 폴더 만들고 git 저장소로 초기화해줘

# 간단한 파일 만들기
> 기본 "Hello World" 페이지로 index.html 만들어줘

# 첫 커밋 하기
> "Initial commit"으로 커밋해줘

# 상태 확인
> git 상태 보여줘
```

### 연습 2: 변경하고 커밋하기

```
# 파일 수정
> 제목을 "내 첫 Git 프로젝트"로 바꿔줘

# 뭐가 바뀌었는지 보기
> 뭐가 바뀌었는지 보여줘 (git diff)

# 변경사항 커밋
> 이 변경사항 커밋해줘

# 히스토리 보기
> 커밋 히스토리 보여줘
```

### 연습 3: 브랜치 사용해보기

```
# 새 브랜치 만들기
> feature/new-header라는 브랜치 만들어줘

# 브랜치에서 변경하기
> 내 이름이 있는 헤더 추가해줘

# 커밋하기
> 커밋해줘

# main으로 돌아가기
> main 브랜치로 전환해줘

# 헤더가 없는 것 확인 (다른 브랜치에 있으니까)
> 헤더 있어?
```

## 문제가 발생하면

**"not a git repository" 에러가 나옵니까?**
- Git 폴더에 있지 않습니다. 먼저 초기화하십시오:
- Claude에게: "이 폴더에 git 초기화해줘"

**"nothing to commit" 메시지가 나옵니까?**
- 마지막 커밋 이후 파일을 변경하지 않았습니다
- 먼저 변경하고, 그 다음 커밋하십시오

**머지할 때 "conflict"가 발생합니까?**
- 두 브랜치가 같은 줄을 수정했습니다
- Claude에게: "이 머지 충돌 해결해줘"

**실수로 잘못된 것을 커밋했습니까?**
- 당황하지 마십시오. "마지막 커밋 취소하고 변경사항은 유지해줘"
- 또는: "마지막 5개 커밋 보여줘"로 어디로 돌아갈지 찾기

**푸시하면 안 되는 것을 푸시했습니까?**
- 주의: 이건 되돌리기 어렵습니다
- 민감한 파일은 팀장에게 연락하십시오
- 코드 실수는: 고치는 새 커밋 만들기

**Git 명령어가 헷갈립니까?**
- 외울 필요 없습니다
- Claude에게 원하는 것을 말하십시오: "내 변경사항 보고 싶어" 또는 "작업 저장해줘"

## 자주 하는 실수

### 실수 1: 정기적으로 커밋 안 하기

```
# 나쁜 예 - 며칠 작업 후 하나의 거대한 커밋
> 모든 변경사항 "작업함"으로 커밋해줘

# 좋은 예 - 작고 자주 커밋
> 로그인 폼 커밋해줘
> 검증 커밋해줘
> 에러 처리 커밋해줘
```

### 실수 2: 모호한 커밋 메시지

```
# 나쁜 예
> "고침"으로 커밋해줘

# 좋은 예
> 커밋해줘. 이메일 검증 버그 수정한 거야.
```

### 실수 3: 모든 것을 Main 브랜치에서 작업

```
# 나쁜 예 - 모든 변경을 main에 직접
> [main에서 변경]
> 커밋

# 좋은 예 - 기능에 브랜치 사용
> 새 기능용 브랜치 만들어줘
> [변경]
> 커밋
> main에 머지하는 PR 만들어줘
```

### 실수 4: 커밋 전에 상태 확인 안 하기

```
# 나쁜 예 - 맹목적으로 모든 것 커밋
> 모든 변경사항 커밋해줘

# 좋은 예 - 뭘 커밋하는지 확인
> 어떤 파일이 바뀌었어?
> [목록 검토]
> 로그인 관련 파일만 커밋해줘
```

### "nothing to commit"

커밋할 변경사항이 없습니다. 파일을 수정했는지 확인하십시오.

### "not a git repository"

Git 저장소가 아닙니다. `git init`으로 초기화하십시오.

### 실수로 커밋했을 때

```
> 마지막 커밋 취소해줘 (변경사항은 유지)
```
