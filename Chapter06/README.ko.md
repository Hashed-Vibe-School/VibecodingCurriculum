## 프롬프팅이란?

프롬프팅은 AI에게 요청하는 방법입니다. 같은 내용을 물어봐도 어떻게 표현하느냐에 따라 결과가 달라집니다.

### 나쁜 예 vs 좋은 예

**나쁜 예:**
```
> 버그 고쳐줘
```
Claude가 무엇을 수정해야 하는지 알 수 없습니다.

**좋은 예:**
```
> @src/login.js에서 로그인 버튼 누르면 에러 발생.
> "Cannot read property 'email' of undefined" 에러.
> 수정해줘.
```
Claude가 정확히 무엇을 해야 하는지 파악할 수 있습니다.

## 좋은 프롬프트의 3가지 원칙

### 1. 구체적으로 표현

```
# 나쁜 예
> 웹사이트 만들어줘

# 좋은 예
> 자기소개 페이지 만들어줘.
> 이름, 사진, 소개글 섹션이 있고
> 배경은 파란색으로.
```

### 2. 맥락 제공

```
# 나쁜 예
> 함수 추가해줘

# 좋은 예
> @src/utils.js에 날짜 포맷 함수 추가해줘.
> 다른 함수들과 비슷한 스타일로.
> "2024-01-15"를 "2024년 1월 15일"로 변환하는 함수.
```

### 3. 한 번에 하나씩

```
# 나쁜 예
> 로그인 기능 만들고 회원가입도 만들고 비밀번호 찾기도 만들어줘

# 좋은 예
> 먼저 로그인 기능부터 만들자.
> 이메일과 비밀번호 입력받는 폼.
```

## Plan 모드 활용

Plan 모드는 Claude가 실행하기 전에 계획을 세우게 합니다.

### Plan 모드의 장점

바로 코드를 작성하면 Claude가 잘못된 방향으로 갈 수 있습니다. Plan 모드에서 먼저 계획을 확인하고, 괜찮으면 실행하는 것이 좋습니다.

### 사용법

```
> /plan
```
또는 `Shift + Tab` 두 번

### 예시

```
> /plan
> 로그인 기능 어떻게 만들면 좋을까?
```

Claude가 계획을 표시합니다:
```
로그인 기능 구현 계획:

1. src/components/LoginForm.js 생성
   - 이메일, 비밀번호 입력 필드
   - 제출 버튼

2. src/api/auth.js 생성
   - 로그인 API 호출 함수

3. 기존 App.js 수정
   - 로그인 상태 관리 추가

이렇게 진행할까요?
```

괜찮으면 Normal 모드로 전환해서 실행합니다.

## 단계별 진행

복잡한 작업은 분할해서 진행합니다.

### 나쁜 예

```
> 쇼핑몰 만들어줘
```
요청 범위가 너무 넓습니다.

### 좋은 예

```
# 1단계: 이해
> 쇼핑몰에 보통 어떤 기능이 있어?

# 2단계: 계획
> 그중에서 상품 목록 페이지 먼저 만들자.
> 어떻게 만들면 좋을까?

# 3단계: 실행
> 좋아, 그 계획대로 만들어줘.

# 4단계: 확인
> 잘 되는지 확인하자. 실행해줘.
```

## 피드백 제공

첫 결과가 마음에 들지 않으면 피드백을 제공합니다.

### 예시

```
> 버튼 만들어줘
```
결과가 나왔는데 너무 작다면:

```
> 버튼이 너무 작아. 더 크게 해줘.
> 글자도 더 굵게.
```

계속 다듬어가면 됩니다.

### 되돌리기

완전히 잘못된 방향이면:
- `Esc Esc` (Esc 두 번): 이전 상태로 되돌리기
- `Ctrl + C`: 현재 작업 취소

## Ultrathink: 복잡한 문제 해결

어려운 문제에는 Claude가 더 깊이 생각하게 할 수 있습니다.

### 키워드

| 키워드 | 효과 | 사용 시점 |
|--------|------|-----------|
| `think` | 보통 수준 | 일반적인 작업 |
| `think hard` | 깊이 생각 | 복잡한 문제 |
| `ultrathink` | 최대한 깊이 | 아키텍처 결정 |

### 예시

```
> ultrathink
> 이 프로젝트 구조를 어떻게 설계하면 좋을까?
> 나중에 기능 추가하기 쉽게.
```

```

```

## 심화: 프롬프팅 고급 기법

### 범위를 제한하기

AI는 주어진 자유도가 높을수록 예상과 다른 방향으로 갈 수 있습니다. "버튼 만들어줘"라고 하면 10개의 헬퍼 함수와 3개의 유틸리티 파일이 생길 수도 있습니다.

범위를 명확히 하면 이런 상황을 방지할 수 있습니다:

```
> 버튼 컴포넌트 만들어줘.
> 이 파일 하나에서 해결하고, 추가 파일은 만들지 마.
```

### 배경 정보 제공하기

같은 요청도 맥락에 따라 최적의 해결책이 달라집니다.

| 맥락 없음 | 맥락 있음 |
|----------|----------|
| "이 함수 최적화해줘" | "이 함수는 페이지 로드마다 호출돼서 빨라야 해" |
| "로그인 만들어줘" | "해커톤용 프로토타입이야. 복잡하게 안 해도 돼" |

Claude는 "왜 이걸 하는지"를 알면 상황에 맞는 트레이드오프를 선택할 수 있습니다.

### 결과가 안 좋을 때

원하는 결과가 나오지 않으면 요청을 다시 살펴보세요:

- 너무 모호하지 않았나?
- 중요한 맥락을 빠뜨리지 않았나?
- 한 번에 너무 많은 것을 요청하지 않았나?

요청을 개선하면 결과도 개선됩니다. 이건 모델의 한계가 아니라 커뮤니케이션의 문제인 경우가 많습니다.

### 작업을 나누는 것의 힘

복잡한 시스템은 작은 부품들의 조합입니다. AI 에이전트도 마찬가지로, 작은 단위의 작업을 잘 수행하고 이를 조합해서 복잡한 결과를 만들어냅니다.

큰 작업을 한 번에 요청하면:
- 중간에 잘못되어도 어디가 문제인지 파악하기 어렵습니다
- 방향 수정이 어렵습니다

작은 단위로 나누면:
- 각 단계의 결과를 확인할 수 있습니다
- 문제가 생기면 해당 단계만 다시 하면 됩니다
- 진행 상황이 명확합니다

```
# 한 번에 요청
> 결제 시스템 전체 만들어줘

# 단계별 요청
> 먼저 결제 버튼 UI만 만들어줘
> (확인 후) 결제 API 연동해줘
> (확인 후) 결제 완료 페이지 만들어줘
```

### 점진적 개선 사이클

처음부터 완벽할 필요 없습니다. AI와의 작업은 반복적인 개선 과정입니다:

1. 일단 시도해본다
2. 결과를 확인한다
3. 부족한 부분을 피드백한다
4. 더 나은 결과를 얻는다

이 사이클을 빠르게 돌리는 것이 한 번에 완벽한 요청을 만들려고 고민하는 것보다 효율적입니다.
